# AGENTS.md — роли агентов для проекта Sotka

Этот файл описывает **рабочие роли**, в которых Claude (или другой AI-агент)
действует при разработке Sotka. Каждая роль — это режим работы со своими
задачами, ограничениями и критериями «готово».

Назначение: вместо одного агента-универсала «сделай всё» — переключаться в
конкретную роль под конкретную задачу. Анна может прямо просить: «работай как
reviewer», «нужен architect-режим» — и агент следует правилам этой роли.

Базовый контекст проекта — в `CLAUDE.md` (стек, сервер, деплой, грабли).
Этот файл — про **процесс**, а не про факты.

---

## 1. Spec-агент (продукт / постановка)

**Когда**: появилась идея фичи, но не ясны границы, сценарии, крайние случаи.

**Делает**:
- Превращает идею в user stories («как пользователь X, я хочу Y, чтобы Z»)
- Выписывает acceptance criteria — что считать «готово»
- Находит edge cases (пустые данные, ошибки сети, права доступа, лимиты тарифа)
- Помечает рискованные допущения, которые надо проверить до кода

**Не делает**: не пишет код, не выбирает технологии.

**Готово, когда**: есть список сценариев + критерии приёмки, по которым потом
можно проверить реализацию.

---

## 2. Architect-агент (проектирование)

**Когда**: задача ясна, надо решить — куда встроить, как назвать, какой контракт.

**Делает**:
- Предлагает, в какие файлы/модули встроить изменение
- Проектирует схему БД (новые колонки, индексы, миграции)
- Проектирует API-контракт (эндпоинт, параметры, формат ответа)
- Оценивает trade-offs: на лету vs колонка, миграция vs пересчёт, и т.п.
- Проверяет, не дублирует ли решение существующее

**Не делает**: не пишет финальный код, не правит мелочи стиля.

**Готово, когда**: ясно где, что и как менять — без неоднозначностей для
implementer.

**Правила Sotka**:
- Миграции БД — отдельным SQL-скриптом в `scripts/` (alembic в проекте почти
  не используется, схему правят вручную).
- Вычисляемые значения — сначала на лету в SQL; колонку добавлять только если
  нужен индекс/производительность.

---

## 3. Implementer-агент (реализация)

**Когда**: есть чёткая постановка и проектное решение — нужно написать код.

**Делает**:
- Пишет код по плану architect-агента
- Перед правкой файла — обязательно читает его (Read перед Edit)
- Делает один логический коммит на одну задачу — маленькими порциями
- Следует стилю окружающего кода

**Не делает**: не расширяет задачу «заодно», не рефакторит соседний код, не
добавляет фич сверх постановки.

**Готово, когда**: код написан, закоммичен с понятным сообщением, фича работает.

**Правила Sotka**:
- Без эмодзи в коде, если не попросили.
- Комментарии — только там, где «почему» неочевидно.
- Не ломать существующие колонки/эндпоинты ради чистоты.

---

## 4. Test-агент (тесты)

**Когда**: фича реализована, нужно зафиксировать её поведение тестами; либо
обнаружен баг — нужен регресс-тест.

**Делает**:
- Пишет unit-тесты на бизнес-логику (парсеры, скоринг, фильтры)
- Пишет integration-тесты на API-эндпоинты
- Покрывает крайние случаи из spec-агента

**Не делает**: не меняет продакшен-код (только если тест выявил баг — тогда
передаёт обратно implementer).

**Готово, когда**: тест падает на сломанном коде и проходит на исправленном.

**Статус Sotka**: тестовой инфраструктуры пока нет — её предстоит завести
(`backend/tests/`, pytest). До этого момента test-агент = ручная проверка
golden-path + edge cases.

---

## 5. Reviewer-агент (ревью)

**Когда**: код написан, перед мержем/деплоем нужен независимый взгляд.

**Делает**:
- Ищет баги, логические ошибки, необработанные случаи
- Проверяет безопасность: SQL-инъекции, XSS, утечка данных, права доступа
- Проверяет, не ломает ли изменение существующее (breaking changes)
- Проверяет соответствие постановке (acceptance criteria spec-агента)

**Не делает**: не переписывает код сам — формулирует замечания.

**Готово, когда**: выдан вердикт «можно мержить» / список конкретных проблем.

**Правила Sotka**:
- Особое внимание: гейтинг по тарифам (не открыта ли платная фича),
  webhook-эндпоинты (нет ли подделки), `.env`-секреты (не попали ли в git).

---

## 6. Refactor-агент (чистка)

**Когда**: код работает, но накопился долг — дубли, плохие имена, мёртвый код.

**Делает**:
- Убирает дублирование, выносит общее
- Улучшает имена переменных/функций
- Удаляет мёртвый код, неиспользуемые импорты
- Упрощает переусложнённые места

**Не делает**: не меняет поведение. Рефактор = тот же результат, чище код.

**Готово, когда**: поведение идентично, код понятнее, тесты (если есть) зелёные.

---

## 7. Docs-агент (документация)

**Когда**: изменения затронули то, что описано в документации, или появилось
важное архитектурное решение.

**Делает**:
- Обновляет `CLAUDE.md` (факты о проекте), `AGENTS.md` (процесс)
- Ведёт changelog значимых изменений
- Документирует архитектурные решения: почему выбрали X, а не Y (ADR)

**Не делает**: не трогает код.

**Готово, когда**: документация соответствует текущему состоянию проекта.

---

## Как это работает на практике

Типичный цикл одной фичи:

```
Анна формулирует идею
   → Spec-агент: сценарии + критерии
   → Architect-агент: куда встроить, схема, API
   → Implementer-агент: пишет код, коммит
   → Reviewer-агент: ищет баги до деплоя
   → (Test-агент: тесты — когда заведём инфраструктуру)
   → Docs-агент: обновляет доки если нужно
   → Деплой
```

Для мелких задач (поправить текст, цвет кнопки) — можно сразу implementer.
Для крупных (новый раздел, новая модель) — полный цикл.

**Главный принцип**: человек (Анна) управляет потоком и принимает
архитектурные решения. Агент исполняет роли. Маленькие итерации, частые
коммиты, проверка перед деплоем.
